# 10 — 面向未来的 Agent 范式：有状态 AI 实体

> 从"聊天 + 工具"到"持续运行、感知环境、积累经验、主动行动"的 AI 实体。
>
> **注**: 本文档为完整思考过程记录。意图层已归档为远期探索。当前 MVP 已落地行动层、记忆层和 daemon/TUI 架构；感知层 Timer/Watcher/Webhook/Stream 仍是预留设计。

## 为什么现有范式到达天花板

所有现有 agent（Claude Code / OpenClaw / Codex / 所有 Claw 变体）共享同一个架构：

```
用户消息 → 构建 Prompt (注入记忆) → LLM 推理 → 工具调用 → 回复 → 等待
```

这个架构的根本缺陷：

**1. 无状态循环**
- 每一轮都是独立事件
- 记忆只是"上一轮的附注"
- Agent 不在"活着"，只是被反复"唤醒"

**2. 线性上下文**
- 对话历史 = 一维数组
- 长任务越做越慢
- 压缩/摘要是在打补丁，不是解决方案

**3. 被动响应**
- Agent 永远在等用户
- 不会主动观察、预测、准备
- "学习"只是记住事实，不是积累经验

**4. 上下文丢失**
- OpenClaw 真实事故：压缩后丢失指令，agent 失控删邮箱
- 本质原因：指令和对话历史混在同一个线性序列里
- 压缩时无法区分"关键指令"和"可丢弃的中间过程"

## 新范式：有状态 AI 实体

### 核心隐喻

```
传统 Agent = 算命先生
  你说一句，他答一句，说完就忘了

有状态实体 = 学徒
  持续在场，观察环境，积累经验
  你不说话他也在观察和学习
  越久越懂你
```

### 架构对比

```
传统架构:
  ┌──────────────────────────────────┐
  │  Agent Loop (无状态)              │
  │                                    │
  │  用户消息 → LLM → 工具 → 回复     │

  └──────────────────────────────────┘

新架构:
  ┌──────────────────────────────────────────────────┐
  │  sunad (守护进程，常驻)                            │
  │                                                    │
  │  ┌──────────┐  ┌──────────┐                       │
  │  │ 感知层    │  │ 记忆层    │                       │
  │  │ Sense    │  │ Memory   │                       │
  │  │          │  │          │                       │
  │  │ 环境     │  │ Active   │                       │
  │  │ 文件变化  │→│ Memory   │                       │
  │  │ 消息     │  │ 异步整理  │                       │

  │  │ 行动层 Act                                    │  │
```

## 详细设计（三层 + 意图归档）

注：以下为完整思考记录。MVP 实现三层（感知/记忆/行动），意图层归档为远期探索。

### Layer 1: 感知层 (Sense)

传统 agent 的感知是被动的——只有当用户发消息或触发器触发时才"醒来"。

有状态实体的感知是主动的——持续监听环境变化。

#### 感知源

```
1. 用户消息       → 直接指令 (传统 agent 只有这个)
2. 文件变化       → fsnotify 监听工作目录
3. 命令输出流     → Stream 触发器
4. 时间事件       → Cron 定时
5. 外部 Webhook   → HTTP 端点
6. 习惯信号       → 内部生成 (见意图层)
```

#### 感知过滤器

不是所有环境变化都需要 agent 处理：

```
感知到的信号 → 过滤器 → 决定是否需要处理

过滤规则:
  - 用户直接消息 → 总是处理
  - 文件变化:
    - .git/ 变化 → 忽略
    - node_modules/ 变化 → 忽略
    - .go 文件保存 → 检查是否需要自动跑测试 (如果有这个习惯)
    - config 文件变化 → 检查是否需要重载
  - 时间事件:
    - 每天早上9点 → 如果有"早报"习惯 → 准备日报
    - 否则 → 忽略

过滤器由意图层动态配置，不是写死的规则。
```

#### 感知的节流

```
环境变化可能很频繁（保存一次文件可能触发 5 个 fsnotify 事件）

策略:
  - Debounce: 2秒内的同类事件合并
  - 优先级: 用户消息 > 时间事件 > 文件变化 > 流数据
  - 意图驱动: 只有和当前意图相关的感知才进入处理流程
```

### Layer 2: 记忆层 (Memory)

当前记忆层已收敛为轻量 active memory。目标不是保存完整历史，而是让 Suna 越用越懂用户。

#### 设计原则

```
原则 1: 不使用 embedding，不依赖不稳定外部服务。
原则 2: 不保存完整会话历史，只保留最近会话恢复快照。
原则 3: 不做 append-only，记忆会刷新、合并、替换、删除。
原则 4: 只记录用户偏好、习惯、性格、长期约束、纠错和少量事实。
原则 5: 主链路只读记忆，daemon 异步 full compaction。
```

#### 记忆结构

```
user_memory:
  长期 active memory，默认最多 30 条。

conversation_state:
  最近一轮恢复摘要和 last_messages，用于退出后恢复。

memory_queue:
  临时队列，daemon 批量处理后删除。
```

#### 记忆整理

```
old_user_memory + new_queue_events -> new_user_memory
```

daemon 使用 LLM 输出完整的新 active memory 列表，系统代码 diff 后更新数据库。这样天然支持替换旧偏好、删除过期信息和合并重复记忆。

### Layer 3: 意图层 (Intent) — ⚠️ 已归档，MVP 不实现

这是新范式最核心的创新。传统 agent 没有意图层——它只是响应。

#### 意图 ≠ 任务

```
任务 (Task):  具体的、一次性的操作
  例: "帮我重构 auth 模块"

意图 (Intent): 持续的、方向性的驱动力
  例: "保持代码质量" → 驱动 agent 在改完代码后自动跑测试
  例: "不遗漏重要邮件" → 驱动 agent 主动检查和提醒
```

#### 意图的来源

```
1. 显式意图: 用户明确要求的
   "每天早上给我汇总邮件"
   → 意图: 邮件汇总 (驱动: 定时触发)

2. 隐式意图: 从行为模式中识别的
   用户连续 5 天在早上 9 点问 "昨天有什么重要邮件"
   → 意图: 邮件关注 (驱动: 习惯信号)
   → agent 主动: "我注意到你每天都问邮件，要不要我每天早上自动汇总？"

3. 学习意图: 从失败/纠正中提炼的
   用户连续 3 次纠正 "代码要跑 go test"
   → 意图: 代码质量 (驱动: 失败记忆)
   → agent 以后改 .go 文件后自动跑测试

4. 项目意图: 从项目配置中读取的
   AGENTS.md: "使用 golangci-lint"
   → 意图: 代码规范 (驱动: 项目配置)
```

#### 意图的生命周期

```
  识别 → 确认 → 激活 → 执行 → 休眠 → 重新激活

识别:
  感知层检测到模式 → 提交给意图层
  例: 用户第 3 次在某类任务后跑测试

确认:
  Agent 询问用户: "我注意到你总是在改代码后跑测试，要不要我自动做？"
  用户确认 → 意图激活
  用户拒绝 → 意图标记为"不感兴趣"，不再询问

激活:
  意图变为活跃状态，开始驱动感知和行动
  例: "代码质量" 意图激活后:
    - 感知层: 监听 .go 文件保存事件
    - 行动层: 保存后自动跑 go test

执行:
  意图触发具体行动
  执行结果反馈到 memory_queue，由 daemon 判断是否更新 active memory

休眠:
  意图 30 天未被触发 → 降级为休眠
  休眠意图仍保留，不删除
  再次检测到相关模式 → 重新激活

意图是否保留由后续设计决定；active memory 本身不是仅添加式。
```

#### 意图的优先级

```
当多个意图同时需要行动时:

  用户直接消息 > 时间事件 (定时任务) > 习惯信号 > 项目意图

例:
  早上 9:00:
    - 定时意图: 汇总邮件 (优先级: 中)
    - 用户发消息: "帮我修个 bug" (优先级: 高)
  → 先处理用户消息，邮件汇总排队

  文件保存:
    - 习惯意图: 跑测试 (优先级: 低)
    - 用户正在对话中 (优先级: 高)
  → 不打断对话，测试排队到空闲时执行
```

### Layer 4: 行动层 (Act)

行动层和原设计基本一致，但有一个关键区别：**行动不只由 LLM 驱动，也可以由意图层直接驱动。**

```
传统: 用户消息 → LLM → 工具调用
新:   意图触发 → 判断是否需要 LLM → 直接执行 or LLM 规划

例 1 (直接执行):
  意图: "代码质量" 感知到 .go 文件保存
  → 直接执行: go test ./... (不需要 LLM，确定性逻辑)

例 2 (LLM 规划):
  意图: "邮件汇总" 触发
  → LLM 规划: 读取邮件 → 筛选重要 → 生成摘要 → 发送

直接执行 vs LLM 规划的判断:
  - 习惯性的、确定性的操作 → 直接执行 (省 token，低延迟)
  - 需要理解、判断、创作的操作 → LLM 规划 (需要智能)
  - 不确定 → LLM 规划 (宁可多花 token，不要误操作)
```

#### 渐进信任

和原设计一致，但和意图层联动：

```
意图确认 = 天然的高信任度
  用户确认了"改代码后跑测试"的意图
  → 该意图下的 go test 操作自动获得高信任度
  → 不需要每次都过 Guard

但其他操作仍然走 Guard:
  意图: "代码质量" 触发 go test → 高信任，直接执行
  同一个意图触发 go test 后发现失败
  → agent 想自动修改代码修复 → 这是新的操作，走 Guard
```

## 与原 Suna 设计的关系

```
原设计的 7 大模块 → 新范式的 3 层 + 意图探索

原: Agent 内核     → 新: 3 层架构 (感知/记忆/行动)
原: 多模型路由     → 新: 行动层的一部分 (模型选择由记忆驱动)
原: 核心工具       → 新: 行动层 (不变，9 工具 + skill + MCP)
原: 权限守卫       → 新: 行动层的一部分 (渐进信任，意图信任待探索)
原: Skill / MCP     → 新: 行动层的一部分 (通用 Skill + MCP 独立配置)
原: 分层记忆       → 新: 记忆层 (轻量 active memory)
原: 事件触发       → 新: 感知层 (触发器变成感知源)
新: 意图层         → 归档为远期探索
```

## 关键差异化总结

```
1. 有状态实体 (vs 无状态循环)
   - 持续运行，不只是被唤醒
   - 感知环境，不只是等消息

2. 轻量 active memory (vs 线性对话 + 压缩)
   - 只保留少量有效用户画像
   - 记忆会刷新、合并、替换和删除
   - 动态上下文短，利于 prompt cache

3. 主动感知 (vs 纯响应式)
   - 4 种感知源持续监听环境
   - 感知信号直接驱动行动层

4. Go 单二进制 + 多模型 (不变)
   - 仍然是 Suna 的技术差异化

5. 意图层 (归档为远期探索)
   - 主动预测和准备
   - 从习惯中学习
   - 意图驱动的直接执行
```

## 实现优先级

```
Phase 1 (5周): Daemon 基础 + 记忆层基础
  - sunad 守护进程 + IPC (Unix Socket + JSON-RPC)
  - 9 tool definitions + Guard stub
  - active memory + memory_queue + conversation_state

Phase 2 (4周): 行动层完善
  - Guard 完善 + 多模型路由
  - 渐进信任和感知源 (Timer/Watcher/Webhook/Stream) 仍为后续项

Phase 3 (4周): 记忆深化 + 学习
  - active memory 质量评估 + 用户可编辑记忆
  - 通用 Skill 导入/check/启用与按需加载
  - MCP 独立配置与接入

Phase 4 (4周): 完善 + 扩展
  - 项目配置 (AGENTS.md)
  - 模型表现追踪 + 最近会话恢复

Phase 5 (远期): 意图层探索
  - 习惯识别
  - 意图确认 + 激活
  - 意图信任
  - 意图驱动的直接执行
  - WebSocket Transport (远程客户端)
```

## 开放问题

```
1. ✅ 已解决: 不做向量索引
   决策: active memory 数量很小，规则召回足够 (见 06-memory.md)

2. 意图识别的可靠性 (远期，依赖意图层)
   习惯模式识别需要多少样本？
   误识别怎么办？(频繁询问用户会很烦)
   需要设计"不打扰"的阈值

3. active memory 的质量
   长期使用后需要避免过度泛化、误删和无效记忆
   当前策略: full compaction + 30 条上限 + core memory 上限
   远期: 设计用户可编辑、可删除、可纠正的记忆 UI

4. 感知层的资源消耗
   持续监听文件/网络需要系统资源
   需要设计合理的资源预算
   笔记本电脑上不能太耗电
```
