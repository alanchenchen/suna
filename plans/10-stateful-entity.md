# 10 — 面向未来的 Agent 范式：有状态 AI 实体

> 从"聊天 + 工具"到"持续运行、感知环境、积累经验、主动行动"的 AI 实体。
>
> **注**: 本文档为完整思考过程记录。意图层已归档为远期探索，MVP 实现三层架构（感知 + 记忆 + 行动）。

## 为什么现有范式到达天花板

所有现有 agent（Claude Code / OpenClaw / Codex / 所有 Claw 变体）共享同一个架构：

```
用户消息 → 构建 Prompt (注入记忆) → LLM 推理 → 工具调用 → 回复 → 等待
```

这个架构的根本缺陷：

**1. 无状态循环**
- 每一轮都是独立事件
- 记忆只是"上一轮的附注"
- Agent 不在"活着"，只是被反复"唤醒"

**2. 线性上下文**
- 对话历史 = 一维数组
- 长任务越做越慢
- 压缩/摘要是在打补丁，不是解决方案

**3. 被动响应**
- Agent 永远在等用户
- 不会主动观察、预测、准备
- "学习"只是记住事实，不是积累经验

**4. 上下文丢失**
- OpenClaw 真实事故：压缩后丢失指令，agent 失控删邮箱
- 本质原因：指令和对话历史混在同一个线性序列里
- 压缩时无法区分"关键指令"和"可丢弃的中间过程"

## 新范式：有状态 AI 实体

### 核心隐喻

```
传统 Agent = 算命先生
  你说一句，他答一句，说完就忘了

有状态实体 = 学徒
  持续在场，观察环境，积累经验
  你不说话他也在观察和学习
  越久越懂你
```

### 架构对比

## 详细设计（三层 + 意图归档）

注：以下为完整思考记录。MVP 实现三层（感知/记忆/行动），意图层归档为远期探索。

### Layer 1: 感知层 (Sense)

传统 agent 的感知是被动的——只有当用户发消息或触发器触发时才"醒来"。

有状态实体的感知是主动的——持续监听环境变化。

#### 感知源

```
1. 用户消息       → 直接指令 (传统 agent 只有这个)
2. 文件变化       → fsnotify 监听工作目录
3. 命令输出流     → Stream 触发器
4. 时间事件       → Cron 定时
5. 外部 Webhook   → HTTP 端点
6. 习惯信号       → 内部生成 (见意图层)
```

#### 感知过滤器

不是所有环境变化都需要 agent 处理：

```
感知到的信号 → 过滤器 → 决定是否需要处理

过滤规则:
  - 用户直接消息 → 总是处理
  - 文件变化:
    - .git/ 变化 → 忽略
    - node_modules/ 变化 → 忽略
    - .go 文件保存 → 检查是否需要自动跑测试 (如果有这个习惯)
    - config 文件变化 → 检查是否需要重载
  - 时间事件:
    - 每天早上9点 → 如果有"早报"习惯 → 准备日报
    - 否则 → 忽略

过滤器由意图层动态配置，不是写死的规则。
```

#### 感知的节流

```
环境变化可能很频繁（保存一次文件可能触发 5 个 fsnotify 事件）

策略:
  - Debounce: 2秒内的同类事件合并
  - 优先级: 用户消息 > 时间事件 > 文件变化 > 流数据
  - 意图驱动: 只有和当前意图相关的感知才进入处理流程
```

### Layer 2: 记忆层 (Memory)

这是和现有设计差异最大的部分。借鉴 Mem0 最新研究，重新设计。

#### 设计原则

```
原则 1: 记忆不是对话历史的附件，而是独立的层次化知识库
原则 2: 新旧事实共存，不做覆盖/删除 (仅添加式提取)
原则 3: Agent 生成的信息是第一类的 (不只是记住用户说的)
原则 4: 检索是多信号的 (语义 + 关键词 + 实体)
原则 5: 时间推理是内置的 (知道"之前"和"现在"的关系)
```

#### 记忆层次

```
┌─────────────────────────────────────────────────────────────┐
│ 工作记忆 (Working Memory)                                     │
│ 存储: 进程内                                                   │
│ 生命周期: 当前任务                                              │
│ 内容: 当前任务的上下文、正在处理的文件、活跃的工具调用链          │
│ 特点: 快速读写，任务完成后归档到长期记忆                        │
├─────────────────────────────────────────────────────────────┤
│ 情景记忆 (Episodic Memory)                                    │
│ 存储: SQLite + 向量索引                                        │
│ 生命周期: 永久 (30天未使用降级)                                 │
│ 内容: 每次交互的片段——谁说了什么、做了什么、结果如何             │
│ 特点: 仅添加式提取，不做覆盖；支持时间线回溯                    │
│ 检索: 语义 + 关键词 + 实体 + 时间范围                          │
├─────────────────────────────────────────────────────────────┤
│ 语义记忆 (Semantic Memory)                                     │
│ 存储: SQLite (结构化) + 文件系统 (SKILL.md)                    │
│ 生命周期: 永久                                                  │
│ 内容: 从情景记忆中提取的结构化知识                              │
│       - 用户偏好、习惯、约束                                    │
│       - 技能定义 (SKILL.md + main.js + mcp.json)              │
│       - 项目知识 (架构、技术栈、约定)                            │
│       - 模型表现记录 (哪个模型在什么任务上表现如何)              │
│ 特点: 仅添加式提取；新旧事实共存，用时间戳区分                  │
│       例: "用户住在北京" (2025-05) + "用户搬到上海" (2026-03)   │
│       → agent 知道"旧住址是北京，现住址是上海"                  │
├─────────────────────────────────────────────────────────────┤
│ 程序记忆 (Procedural Memory)                                   │
│ 存储: 文件系统 (capabilities/)                                  │
│ 生命周期: 永久                                                  │
│ 内容: agent 学会的"怎么做"——技能和操作模式                      │
│       - SKILL.md (怎么做某类事)                                 │
│       - main.js (确定性逻辑)                                   │
│       - mcp.json (外部服务集成)                                │
│ 特点: 和语义记忆的区别是"知道什么"vs"知道怎么做"               │
│       由学习流程自动从情景记忆中提炼                            │
└─────────────────────────────────────────────────────────────┘
```

#### 记忆提取 (Extraction)

借鉴 Mem0 的仅添加式提取：

```
每次交互后，自动从对话中提取记忆:

输入: 最后一轮的用户消息 + agent 回复 + 工具调用结果
输出: 一组记忆片段 (facts)

提取方式: 单次 LLM 调用 (fast 模型)
  prompt = "从以下交互中提取所有值得记住的事实，包括:
    - 用户明确说的偏好/约束
    - 用户隐含的需求模式
    - Agent 完成的操作和结果
    - 决策及其原因
    - 错误和教训"

示例:
  用户: "我不用 YAML，配置都用 TOML"
  Agent: "好的，以后配置都用 TOML"
  提取:
    - { type: "preference", key: "config_format", value: "TOML", 
        source: "user_stated", ts: "2026-05-06T10:00" }
    - { type: "action", key: "acknowledged", value: "config_format=TOML",
        source: "agent_confirmed", ts: "2026-05-06T10:00" }

注意: agent 确认的信息也被记录 (agent-generated facts are first-class)

不做的事情:
  ❌ 不和旧记忆做 diff/reconcile (不做 UPDATE/DELETE)
  ❌ 不覆盖旧事实
  ❌ 新事实直接追加

检索时:
  查询 "用户喜欢什么配置格式"
  → 命中两条: "YAML" (旧) 和 "TOML" (新)
  → 按时间排序，取最新的
  → 但旧的不删，因为可能需要 "之前为什么用 YAML"
```

#### 实体关联 (Entity Linking)

```
每个记忆片段自动提取实体:

  "用户在用 Vue3 + Vite 做前端项目"
  → 实体: Vue3, Vite, 前端

  "部署到 staging 时 npm build 失败了，因为缺少 ENV_VAR"
  → 实体: staging, npm, ENV_VAR

实体存储在独立索引中:
  entity_index:
    Vue3 → [memory_12, memory_45, memory_78]
    Vite → [memory_12, memory_56]

检索时: 实体匹配给相关记忆加权
  查询 "前端用什么框架" → 命中实体 Vue3 → 提升相关记忆的排名
```

#### 多信号检索

```
查询: "用户之前怎么处理部署失败的"

三个信号并行:
  1. 语义相似度: embedding(query) vs embedding(memories)
     → 找到 "部署相关的记忆"

  2. 关键词匹配: "部署" "失败" 在记忆文本中的出现
     → 精确匹配关键词

  3. 实体匹配: 查询中的实体 vs 记忆的实体索引
     → 命中 "staging" "npm" 相关的记忆

融合: 三个信号的分数加权合并 → 排名最高的 top-k 返回

Token 效率: 只返回 top-k (~7K tokens)，不塞全部上下文
```

#### 时间推理

```
这是现有记忆系统最弱的地方，但也是最有价值的。

例子:
  记忆库中有:
    - "用户住在北京" (2025-05)
    - "用户搬到上海" (2026-03)
    - "用户说老邻居很吵" (2025-12)

  查询: "用户说邻居吵是什么时候的事？还住在北京吗？"
  
  时间推理:
    "邻居吵" 发生在 2025-12
    当时住在北京 (2025-05 的记录)
    2026-03 搬到了上海
    → 回答: "当时住在北京，现在已经搬到了上海"

  这不是简单的"取最新"，而是理解时间线的因果关系。
```

### Layer 3: 意图层 (Intent) — ⚠️ 已归档，MVP 不实现

这是新范式最核心的创新。传统 agent 没有意图层——它只是响应。

#### 意图 ≠ 任务

```
任务 (Task):  具体的、一次性的操作
  例: "帮我重构 auth 模块"

意图 (Intent): 持续的、方向性的驱动力
  例: "保持代码质量" → 驱动 agent 在改完代码后自动跑测试
  例: "不遗漏重要邮件" → 驱动 agent 主动检查和提醒
```

#### 意图的来源

```
1. 显式意图: 用户明确要求的
   "每天早上给我汇总邮件"
   → 意图: 邮件汇总 (驱动: 定时触发)

2. 隐式意图: 从行为模式中识别的
   用户连续 5 天在早上 9 点问 "昨天有什么重要邮件"
   → 意图: 邮件关注 (驱动: 习惯信号)
   → agent 主动: "我注意到你每天都问邮件，要不要我每天早上自动汇总？"

3. 学习意图: 从失败/纠正中提炼的
   用户连续 3 次纠正 "代码要跑 go test"
   → 意图: 代码质量 (驱动: 失败记忆)
   → agent 以后改 .go 文件后自动跑测试

4. 项目意图: 从项目配置中读取的
   SUNA.md: "使用 golangci-lint"
   → 意图: 代码规范 (驱动: 项目配置)
```

#### 意图的生命周期

```
  识别 → 确认 → 激活 → 执行 → 休眠 → 重新激活

识别:
  感知层检测到模式 → 提交给意图层
  例: 用户第 3 次在某类任务后跑测试

确认:
  Agent 询问用户: "我注意到你总是在改代码后跑测试，要不要我自动做？"
  用户确认 → 意图激活
  用户拒绝 → 意图标记为"不感兴趣"，不再询问

激活:
  意图变为活跃状态，开始驱动感知和行动
  例: "代码质量" 意图激活后:
    - 感知层: 监听 .go 文件保存事件
    - 行动层: 保存后自动跑 go test

执行:
  意图触发具体行动
  执行结果反馈到记忆层 (情景记忆 + 语义记忆)

休眠:
  意图 30 天未被触发 → 降级为休眠
  休眠意图仍保留，不删除
  再次检测到相关模式 → 重新激活

永不删除意图——和记忆一样，仅添加式。
```

#### 意图的优先级

```
当多个意图同时需要行动时:

  用户直接消息 > 时间事件 (定时任务) > 习惯信号 > 项目意图

例:
  早上 9:00:
    - 定时意图: 汇总邮件 (优先级: 中)
    - 用户发消息: "帮我修个 bug" (优先级: 高)
  → 先处理用户消息，邮件汇总排队

  文件保存:
    - 习惯意图: 跑测试 (优先级: 低)
    - 用户正在对话中 (优先级: 高)
  → 不打断对话，测试排队到空闲时执行
```

### Layer 4: 行动层 (Act)

行动层和原设计基本一致，但有一个关键区别：**行动不只由 LLM 驱动，也可以由意图层直接驱动。**

```
传统: 用户消息 → LLM → 工具调用
新:   意图触发 → 判断是否需要 LLM → 直接执行 or LLM 规划

例 1 (直接执行):
  意图: "代码质量" 感知到 .go 文件保存
  → 直接执行: go test ./... (不需要 LLM，确定性逻辑)

例 2 (LLM 规划):
  意图: "邮件汇总" 触发
  → LLM 规划: 读取邮件 → 筛选重要 → 生成摘要 → 发送

直接执行 vs LLM 规划的判断:
  - 习惯性的、确定性的操作 → 直接执行 (省 token，低延迟)
  - 需要理解、判断、创作的操作 → LLM 规划 (需要智能)
  - 不确定 → LLM 规划 (宁可多花 token，不要误操作)
```

#### 渐进信任

和原设计一致，但和意图层联动：

```
意图确认 = 天然的高信任度
  用户确认了"改代码后跑测试"的意图
  → 该意图下的 go test 操作自动获得高信任度
  → 不需要每次都过 Guard

但其他操作仍然走 Guard:
  意图: "代码质量" 触发 go test → 高信任，直接执行
  同一个意图触发 go test 后发现失败
  → agent 想自动修改代码修复 → 这是新的操作，走 Guard
```

## 与原 Suna 设计的关系

```
原设计的 7 大模块 → 新范式的 3 层 + 意图探索

原: Agent 内核     → 新: 3 层架构 (感知/记忆/行动)
原: 多模型路由     → 新: 行动层的一部分 (模型选择由记忆驱动)
原: 核心工具       → 新: 行动层 (不变，9 工具 + skill + MCP)
原: 权限守卫       → 新: 行动层的一部分 (渐进信任，意图信任待探索)
原: 能力系统       → 新: 记忆层 (程序记忆)
原: 分层记忆       → 新: 记忆层 (4 层记忆，重新设计)
原: 事件触发       → 新: 感知层 (触发器变成感知源)
新: 意图层         → 归档为远期探索
```

## 关键差异化总结

```
1. 有状态实体 (vs 无状态循环)
   - 持续运行，不只是被唤醒
   - 感知环境，不只是等消息

2. 仅添加式记忆 + 时间推理 (vs 线性对话 + 压缩)
   - 新旧事实共存
   - 理解时间线
   - Agent 生成的事实也是第一类的

3. 主动感知 (vs 纯响应式)
   - 4 种感知源持续监听环境
   - 感知信号直接驱动行动层

4. Go 单二进制 + 多模型 (不变)
   - 仍然是 Suna 的技术差异化

5. 意图层 (归档为远期探索)
   - 主动预测和准备
   - 从习惯中学习
   - 意图驱动的直接执行
```
